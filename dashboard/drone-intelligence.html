Drone Intelligence: Augmented Flood Monitoring

Technical roadmap for evolving Jal Drishti from satellite-based analysis to real-time drone-powered flood monitoring with autonomous feature extraction

Vision & Motivation

Jal Drishti currently relies on satellite-derived data (Copernicus GLO-30 DEM at 30m resolution, Open-Meteo weather, CHIRPS rainfall archives) for flood risk modeling. While effective for regional-scale analysis, this approach has inherent limitations in temporal resolution (6–24hr update cycles), spatial resolution (30m DEM, ~5km rainfall grids), and cloud cover interference.

The Drone Intelligence initiative introduces autonomous survey drones as a complementary data layer — providing real-time, sub-centimeter ground truth during active flood events. This creates a satellite + drone fusion architecture that combines broad regional coverage with pinpoint local accuracy.

Key Objective:

Achieve 5cm Ground Sampling Distance with <5 minute data latency, enabling real-time flood depth measurement, damage assessment, and rescue coordination during active events.

Aerodynamic Analysis

Thrust & Hover Performance

The propulsion system is designed with a 2.8:1 thrust-to-weight ratio, providing ample margin for payload and wind resistance:

Thrust per motor: T = C_t × ρ × n² × D⁴
C_t = 0.012 (thrust coefficient), ρ = 1.225 kg/m³ (air density at sea level), n = rev/s, D = 0.381m (15" prop diameter)
Hover power (ideal): P_hover = W^(3/2) / √(2 × ρ × A_disc)
W = 44.1N (4.5kg MTOW), A_disc = 4 × π × (0.19m)² = 0.454 m². Disc loading = 97.1 N/m². P_hover ≈ 380W total → ~17A from 6S battery at hover.
Forward flight drag: D = ½ × ρ × V² × C_d × A_frontal
At cruise V = 15 m/s: C_d = 0.25, A = 0.08 m², Drag ≈ 1.38N, Parasite power ≈ 20.7W. Total cruise power ≈ 420W.
Endurance Calculation:

Battery energy = 22.2V × 10Ah = 222 Wh. At ~420W cruise power (with payload), usable capacity (80% DoD) = 177.6 Wh → ~25 min cruise + 7 min hover reserve = 32 min mission time.

Stability & Control

The Pixhawk 6X uses cascaded PID loops for attitude stabilization:

Torque equation: τ = I × α = r × ΔF
Where I = moment of inertia (~0.02 kg·m²), r = motor arm length (0.325m), ΔF = differential thrust. Roll/Pitch rate: 300°/s, Yaw rate: 150°/s.

Feature Extraction Pipeline

6-Stage Processing Pipeline

Multi-stage computer vision pipeline running as a hybrid edge-cloud architecture. Stages 1–4 execute on the Jetson Orin NX (onboard), stages 5–6 on the cloud backend:

  1. Raw Data Acquisition: Synchronized capture of RGB (24MP), multispectral (5-band), thermal (640×512), and LiDAR (200k pts/sec) at 5cm GSD. All frames geotagged with RTK-corrected coordinates (±2cm).
  2. Edge Preprocessing: Real-time image stitching (OpenCV), radiometric calibration, and NDWI computation. NDWI = (Green − NIR) / (Green + NIR). Pixels with NDWI > 0.3 flagged as water. Runs at 30 FPS on Jetson Orin NX.
  3. Water Surface Segmentation: Segment Anything Model (SAM) with flood-specific prompts segments water extent. U-Net fallback for edge cases. Combined with NDWI thresholding for 95%+ water detection accuracy across varied conditions.
  4. Object Detection: YOLOv8-X (TensorRT optimized) detects: stranded people, submerged vehicles, damaged structures, debris fields, blocked roads. Custom model trained on 10,000+ labeled flood images. Inference at 45 FPS.
  5. DEM Reconstruction: LiDAR point cloud → Open3D → real-time 5cm DEM. Flood depth = current water surface elevation − pre-flood DEM. Stereo photogrammetry as fallback where LiDAR has gaps.
  6. Fusion & Broadcast: All features encoded as GeoJSON FeatureCollections. Pushed to Jal Drishti dashboard via WebSocket. Drone flood layers rendered on MapLibre alongside existing satellite risk grid.

Frontend Modifications to enhanced.js

The existing dashboard JavaScript receives modifications to consume drone data:

  1. WebSocket Client: New DroneDataManager class connects to wss://host/ws/drone-feed, handles reconnection, and buffers incoming GeoJSON frames.
  2. MapLibre Layers: Three new layers added — drone-flood-fill (water extent), drone-depth-heatmap (depth gradient), drone-detections (YOLO bounding boxes).
  3. Risk Engine Update: The existing calculateRiskScore() function now accepts an optional droneDepth parameter that overrides simulated depth when live drone data is available.
  4. Rescue Path Recalculation: A* pathfinding recalculates in real-time when drone data updates flood depth values, providing ground-truth-based rescue routes.

Implementation Roadmap

Phase 1: Foundation (M1–M3)

Quadcopter assembly, ArduPilot config, RTK GPS calibration, IP55 validation, endurance testing — ₹3.5–4.5L

Phase 2: Sensing (M3–M5)

Camera + LiDAR + thermal mount, gimbal integration, sensor time-sync to GPS PPS, reflectance panel calibration — ₹5–7L

Phase 3: Intelligence (M5–M8)

YOLOv8 training (10k+ images), SAM fine-tuning, TensorRT optimization, LiDAR→DEM pipeline, NDWI pipeline — ₹2–3L

Phase 4: Integration (M8–M10)

FastAPI WebSocket backend, feature_fusion.py, PostGIS + Redis setup, MapLibre drone layers, risk engine updates — ₹1–1.5L

Phase 5: Deployment (M10–M12)

Field trials at Meppadi/Darbhanga/Dhemaji, DGCA compliance, auto-deploy testing, operator training — ₹1.5–2L
